# Mobile UNO 游戏 UI 设计 - 需求文档

## 介绍

本文档定义了为 UNO 游戏创建专门移动端界面的需求。目标是提供一个优秀的竖屏用户体验，采用触摸友好的设计，包括扇形手牌展示和隐藏非必要信息的游戏界面。

## 词汇表

- **HandCards**: 玩家的手牌组件
- **GameBoard**: 游戏主界面，显示出牌区和对手信息
- **OpponentArea**: 显示对手玩家信息和卡牌数量的区域
- **Mobile Viewport**: 移动设备的可见区域
- **Touch Target**: 可触摸的交互元素（最小 44px × 44px）
- **Fan Layout**: 扇形排列，模拟真实卡牌游戏
- **Debounce**: 去抖，防止快速重复点击

## 需求

### 需求 1: 移动设备支持

**用户故事**: 作为移动用户，我希望 UNO 游戏能在各种移动设备和浏览器上正常运行，不受限制。

#### 接受标准

1. WHEN 用户在 iOS Safari 中访问游戏 THEN 游戏应正常显示和运行
2. WHEN 用户在 Android Chrome 中访问游戏 THEN 游戏应正常显示和运行
3. WHEN 用户在微信内置浏览器中访问游戏 THEN 游戏应正常显示和运行
4. WHEN 用户在支付宝内置浏览器中访问游戏 THEN 游戏应正常显示和运行
5. WHEN 用户在不同屏幕尺寸的设备上访问 THEN UI 元素应自适应调整

### 需求 2: 竖屏为主布局

**用户故事**: 作为移动用户，我希望 UNO 游戏默认采用竖屏布局，提供最佳的手持体验。

#### 接受标准

1. WHEN 用户启动游戏 THEN 应默认为竖屏模式
2. WHEN 用户在竖屏模式下操作 THEN 游戏内容应完全显示，不需要滚动
3. WHEN 用户将设备旋转为横屏 THEN 系统应有选项切换为横屏模式（非强制）
4. WHEN 用户在竖屏和横屏之间切换 THEN 游戏状态应保持一致
5. WHILE 用户在竖屏模式下 THEN 游戏布局应针对竖屏优化，不显示冗余信息

### 需求 3: 扇形手牌展示

**用户故事**: 作为卡牌游戏玩家，我希望我的手牌以扇形排列显示，就像真实卡牌游戏一样，提供自然的游戏体验。

#### 接受标准

1. WHEN 玩家进入游戏后 THEN 手牌应以扇形排列显示在屏幕底部
2. WHEN 手牌数量增加或减少 THEN 扇形角度应自动调整保持美观
3. WHEN 用户拥有 5-13 张卡牌 THEN 每张卡牌应清晰可见且能点击
4. WHEN 用户点击手牌 THEN 被选中的卡牌应有视觉反馈（如上升、高亮）
5. WHEN 用户尝试触摸卡牌时 THEN 点击目标大小应至少为 44×44px

### 需求 4: 隐藏非必要信息

**用户故事**: 作为移动用户，我希望游戏界面简洁清爽，只显示必要的游戏信息，提高可读性。

#### 接受标准

1. WHEN 游戏进行中 THEN 只显示当前出牌、对手数量、手牌区域和操作按钮
2. WHEN 非核心信息（如玩家昵称详情、历史出牌等）需要显示 THEN 应通过可展开的面板或抽屉菜单呈现
3. IF 用户是游戏房主 THEN 显示简化的房间管理选项
4. WHEN 游戏状态发生变化 THEN 通过清晰的提示（弹窗或横幅）告知用户
5. WHILE 游戏进行中 THEN 显示区域应不被其他 UI 元素遮挡

### 需求 5: 触摸交互优化

**用户故事**: 作为移动用户，我希望所有交互元素都易于点击和操作，响应迅速，没有卡顿感。

#### 接受标准

1. WHEN 用户点击卡牌或按钮 THEN 响应时间应 < 100ms
2. WHEN 用户点击交互元素 THEN 元素大小应至少为 44×44px（符合 iOS HIG 标准）
3. WHEN 用户快速连续点击相同元素 THEN 应防止重复触发（去抖）
4. WHEN 手指离开屏幕 THEN 300ms 延迟应被消除，即时响应
5. WHEN 用户尝试选择卡牌 THEN 应支持单击选择和可选的滑动选择

### 需求 6: 对手信息显示

**用户故事**: 作为玩家，我希望看到对手的基本信息，但不需要太多细节。

#### 接受标准

1. WHEN 游戏进行中 THEN 对手应在屏幕顶部以简化的卡片形式显示
2. FOR 每个对手 THEN 应显示昵称和当前卡牌数量
3. WHEN 对手数量较多（5 人或以上） THEN 应以紧凑的行列布局显示
4. WHEN 对手位置需要显示 THEN 应简洁标注（如"上家"、"下家"）
5. IF 某个对手出牌时 THEN 应有视觉强调

### 需求 7: 弹窗和提示显示

**用户故事**: 作为用户，我希望游戏中的弹窗和提示消息清晰易读，不遮挡重要的游戏内容。

#### 接受标准

1. WHEN 游戏状态需要用户确认 THEN 应以模态弹窗呈现
2. WHEN 弹窗出现 THEN 按钮大小和间距应符合移动端标准
3. WHEN 提示消息出现 THEN 应显示在屏幕上方或下方的非关键区域
4. WHEN 消息展示后 THEN 应自动消失或提供关闭按钮
5. WHILE 弹窗显示时 THEN 应禁用背景交互

### 需求 8: 游戏结束和返回流程

**用户故事**: 作为玩家，我希望游戏结束后能清晰看到结果，并便捷地返回或重新开始。

#### 接受标准

1. WHEN 游戏结束 THEN 应显示游戏结果（排名、得分等）
2. WHEN 游戏结束弹窗出现 THEN 应提供"返回大厅"和"重新开始"按钮
3. WHEN 用户点击返回 THEN 应返回房间列表或主菜单
4. WHEN 用户选择重新开始 THEN 应返回游戏大厅准备新游戏
5. WHEN 用户长时间未操作 THEN 应有自动返回的选项提示

### 需求 9: 响应式断点设计

**用户故事**: 作为开发者和设计师，我希望定义清晰的响应式设计断点，支持各种手机屏幕。

#### 接受标准

1. WHEN 设备宽度 < 480px THEN 采用超小屏幕布局（旧手机）
2. WHEN 设备宽度 480px-640px THEN 采用小屏幕布局（iPhone SE）
3. WHEN 设备宽度 640px-768px THEN 采用中等屏幕布局（iPhone 12 等）
4. WHEN 设备宽度 > 768px THEN 采用大屏幕布局或提示使用默认布局
5. FOR 每个断点 THEN 应测试文字可读性、按钮可点击性和信息完整性

### 需求 10: 性能优化

**用户故事**: 作为用户，我希望游戏加载快速，运行流畅，即使在低网速环境下也能体验良好。

#### 接受标准

1. WHEN 用户首次打开游戏 THEN 首屏加载时间应 < 3 秒
2. WHEN 用户切换卡牌或执行操作 THEN 应保持 60fps 的流畅动画
3. WHEN 用户在低网速环境下玩游戏 THEN 应支持基本功能正常运行
4. WHEN 页面加载时 THEN 关键资源应优先加载，非关键资源延迟加载
5. WHILE 游戏运行中 THEN 内存占用应保持稳定，不出现泄漏

### 需求 11: 移动端玩家限制

**用户故事**: 作为移动端用户，我希望游戏在移动设备上运行流畅，不会因为参加人数过多而导致卡顿。

#### 接受标准

1. WHEN 用户在移动设备上创建游戏房间 THEN 最多允许 8 个玩家加入
2. WHEN 已有 8 个玩家的房间 THEN 后续尝试加入的用户应收到"房间已满"的提示
3. WHEN 尝试从 PC 端加入 9+ 人房间时用户改用移动设备访问 THEN 应提示不支持或建议离开
4. WHEN 房间人数为 9 人及以上 THEN 移动设备访问该房间应显示友好的提示和选项
5. FOR 移动端 8 人游戏 THEN 所有 UI 元素应清晰可见，对手卡片不应重叠

## 非需求

- 不要对 PC 端进行优化（保持现有状态）
- 不要实现全屏 API（仅支持浏览器的自然横屏）
- 不要重写现有的游戏逻辑，仅优化 UI 层

## 约束条件

- 必须保持与现有游戏逻辑的兼容性
- 不能修改数据库结构或 API 接口
- 必须支持所有主流移动浏览器
- 不能强制横屏或全屏模式
